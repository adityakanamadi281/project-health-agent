## Design principle: **compute deterministically, narrate with an LLM**

The single most important architectural decision in this codebase is the separation between
the two things people conflate when they say "AI agent for RAG status":

1. **What the status *is*** — computed in plain Python (`analysis/`), from thresholds in
   `config.py`, over data normalized by `ingestion/excel_loader.py`. Reproducible, auditable,
   unit-testable, and stable across runs regardless of which LLM (or none) is configured.
2. **How the status is *explained*** — generated by an LLM (`llm/fireworks_client.py` +
   `llm/narrative.py`) that is given the already-computed verdict and told, explicitly, never
   to change it — only to narrate it in plain English for a VP audience. If the LLM is
   unavailable (no `FIREWORKS_API_KEY`, network failure, malformed response), a deterministic
   template built from the same structured reasons is used instead, and every report/deck
   discloses which path produced its narrative (`narrative_source: llm | template-fallback`).

This means the agent **never silently guesses** a project's health from vibes, and it never
**breaks** just because a key is missing or a request times out — both were explicit
requirements ("handles incomplete or messy data gracefully").

## Module map

| Layer | Module | Responsibility |
|---|---|---|
| Entry points | `cli.py` | Typer CLI: `run-weekly`, `run-monthly`, `schedule-weekly` |
| | `scheduler/weekly_job.py` | APScheduler cron wrapper around `run-weekly` (bonus deliverable) |
| Ingestion | `ingestion/excel_loader.py` | Header-name-driven parsing of Smartsheet-style `.xlsx` exports into normalized models; tolerant of `#UNPARSEABLE`, missing columns, missing sheets |
| Model | `models.py` | Vendor-agnostic pydantic models (`ProjectPlan`, `Task`, `ProjectSummary`, `Comment`, `DataQualityNote`) that all downstream code depends on instead of raw spreadsheet cells |
| Analysis | `analysis/signals.py` | Computes concrete signals: schedule variance/overdue counts, milestone pacing, blockers (unmet predecessors, client-stalled status), critical path risk, stakeholder sentiment, and perception gap |
| | `analysis/rag_engine.py` | Deterministic Red/Amber/Green scoring per dimension, overall weighted composite score, and overrides to Red — see [`docs/RAG_METHODOLOGY.md`](docs/RAG_METHODOLOGY.md) |
| LLM | `llm/fireworks_client.py` | Thin `httpx` wrapper around the Fireworks AI chat-completions endpoint, with retries and a `safe_chat()` helper that never raises |
| | `llm/narrative.py` | Prompts Fireworks to turn a `RAGResult` into a plain-English paragraph; deterministic fallback if unavailable |
| Reporting | `reporting/weekly_report.py` | Orchestrates one project plan end-to-end into a `WeeklyReport`; writes Markdown + JSON |
| | `reporting/history.py` | Append-only `outputs/history.jsonl` so trend detection is based on real recorded runs, never fabricated |
| | `reporting/monthly_synthesis.py` | Computes cross-project portfolio statistics in Python, detects week-over-week movement, then asks Fireworks to turn those *statistics* (not raw data) into headline/trends/risks/recommendations |
| Presentation | `presentation/deck_builder.py` | Builds the 6-slide executive `.pptx` from a `MonthlySynthesis` using `python-pptx` (native chart + table, no LLM involved at this stage) |
| Config | `config.py` | All tunables (Fireworks model/URL/key, RAG thresholds, I/O paths) come from environment variables / `.env` via `pydantic-settings` — nothing is hardcoded in code |

## Why Fireworks is called *twice*, never for the verdict itself

1. **Weekly narrative** (`llm/narrative.py`): one call per project, given the computed
   `RAGResult` — produces the "Plain-English Summary" paragraph in each weekly report.
2. **Monthly synthesis** (`reporting/monthly_synthesis.py`): one call per month, given computed
   portfolio statistics (RAG distribution, average variance, blocker themes, sentiment,
   perception gaps, week-over-week movement) — produces the headline, trend bullets,
   emerging risks, and recommendations for the executive deck.

In both cases the LLM's input is already-computed, structured JSON — never the raw spreadsheet
— so it cannot introduce a fact that isn't in the data, and a hallucinated number would be
immediately visible as inconsistent with the deck's own chart/table (which are rendered
straight from the same structured data, bypassing the LLM entirely).

## Data flow, end to end

1. `run-weekly` discovers every `*.xlsx` in `DATA_INPUT_DIR` (default `data/sample_plans/`).
2. Each file → `excel_loader.load_project_plan()` → a `ProjectPlan`.
3. `signals.compute_signals()` turns that into a `ProjectSignals` object.
4. `rag_engine.determine_rag()` scores it into a `RAGResult` (color + reasons, no LLM).
5. `narrative.generate_narrative()` turns the `RAGResult` into prose (LLM or fallback).
6. `weekly_report.generate_weekly_report()` assembles both into a `WeeklyReport`, saved as
   `outputs/weekly/<date>/<Project>.md` + `.json`, and appended to `outputs/history.jsonl`.
7. `run-monthly` re-runs steps 1–6 for every plan, then `monthly_synthesis.build_monthly_synthesis()`
   aggregates all `WeeklyReport`s + history into portfolio stats and an executive narrative,
   and `deck_builder.build_monthly_deck()` renders the 6-slide `.pptx`.

## Extensibility notes

- **Activating the budget/cost signal**: Currently, Budget Burn is scaffolded as `Grey/Unknown` because cost/hours data is not in the source exports. Once a cost column exists, map it in `TASK_HEADER_ALIASES` inside `excel_loader.py`, calculate the burn ratio in `signals.py`, and factor it into the weighted composite score in `rag_engine.py`.
- **Adding a new project plan format**: as long as the new export uses recognizably similar
  header labels, no code changes are needed — the loader matches by label, not position.
- **Swapping the LLM provider**: `fireworks_client.py` is the only file that talks to
  Fireworks; it exposes a plain `chat(system_prompt, user_prompt) -> str` — retargeting to
  another OpenAI-compatible endpoint is a one-file change plus new `.env` values.
